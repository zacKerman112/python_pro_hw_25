1. Поясніть, чому небезпечно залишати DEBUG = True у файлі settings.py
на продакшн-сервері.
Назвіть щонайменше дві основні загрози.

Вмикання цього режиму на робочому (продакшн) сервері може призвести до того що 
злочинці можуть отримати доступ до внутрішніх компонентів системи і це може призвести
до повного взлому або крадіжки даних.


2.Яке призначення змінної ALLOWED_HOSTS і що відбудеться, 
  якщо не додати до неї доменне ім'я PythonAnywhere?

Призначення цієї константи в тому що це список доменних імен або IP-адрес,
я ких мій веб-сервіс зможе приймати запити. 
Вона слугує критичним заходом безпеки для запобігання
HTTP Host header attacks, захищаючи ваш сайт від підміни доменів.

І якщо не додати моє доменне ім'я в PythonAnywhere то я просто отримаю
помилку DisallowedHost і користувач мого сайту побачить сторінку в повідомленням
Invalid HTTP_HOST header і все.


3.Поясніть роль WSGI (Web Server Gateway Interface) у продакшн-середовищі Django. 
Чим він принципово відрізняється від вбудованого сервера python manage.py runserver?

Роль WSGI полягає у тому що він перетворює сирі HTTP-запити від користувача у зручні для Python об'єкти 
і передає їх у Django, а потім переводить відповідь фреймворку назад у формат HTTP.


4. Що робить команда python manage.py collectstatic і чому її потрібно виконати на сервері перед запуском застосунку?
Куди саме Django збирає ці файли після виконання команди (яка змінна у settings.py це визначає)?

Команда python manage.py collectstatic збирає всі статичні файли (CSS, JavaScript, зображення) з усіх додатків проєкту
та папок STATICFILES_DIRS в один уніфікований каталог для подальшого обслуговування вебсервером.
